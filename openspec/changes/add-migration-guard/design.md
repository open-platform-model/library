## Context

`MIGRATIONS.md` is the hand-written home for breaking-change migration recipes (CHANGELOG.md owns per-version summaries, generated by release-please). It drifts because nothing enforces that a breaking change has a recipe. We want a CI gate — "next to release-please" — that blocks when it doesn't. release-please already parses Conventional Commits to find breaking changes and compute the bump; we reuse that signal.

Decisions below were settled in design discussion: **squash-merge (enforced)**, **`Migration: <slug>` trailer** as the key, **breaking-only** scope, **blocking** gate (no auto-graduation yet).

## Goals / Non-Goals

**Goals**
- Block a PR and a release when a breaking change (`type!:` / `BREAKING CHANGE:`) lacks a `MIGRATIONS.md` entry.
- Make detection deterministic and reliable, not heuristic.
- Keep the authoring flow agent-friendly (a `Migration:` line an agent can always emit).

**Non-Goals**
- Auto-graduating `Unreleased` entries into versioned sections at release (deferred; blocking lets a human/agent fix the branch).
- Covering non-breaking `feat:` changes.
- Rolling the gate out to `cli`/`opm-operator`/`core` (prove in `library`, template later).
- Generating migration text automatically.

## Decisions

### D1 — Two signals; only correlation is hard
The job is (1) "is there a breaking change in range?" and (2) "does MIGRATIONS cover it?". (1) is deterministic — Conventional Commit `type(scope)!:` in the subject or a `BREAKING CHANGE:` footer in the body, the same rule release-please uses. (2) needs a **correlation key**.

### D2 — Correlation key: `Migration: <slug>` trailer, unified with the OpenSpec slug
One `<slug>` flows through the OpenSpec change dir (`…-<slug>`), the breaking commit's `Migration: <slug>` trailer, and the `### … — \`<slug>\`` MIGRATIONS entry. Signal (2) becomes set-containment: `{slugs from trailers} ⊆ {slugs in Breaking bucket}`. Escape hatch `Migration: none — <reason>` for breaking-in-name-only changes (keeps the gate trusted by avoiding false-positive blocks).
- **Alternatives rejected:** *coarse "bucket non-empty"* — can't detect stale/mismatched entries (3 breaking commits + 1 stale entry passes). *OpenSpec-archive-slug only* — gaps for breaking changes that don't go through OpenSpec (deps, tiny fixes, the release commit).

### D3 — Squash-merge enforced; trailer rides in `PR_BODY`
Squash discards individual commit bodies, so the trailer only reaches `main` if it is in the squash commit message. Set `squash_merge_commit_message=PR_BODY` (+ `squash_merge_commit_title=PR_TITLE`) so the trailer, placed in the **PR description**, lands on the squashed commit — where `migration-release-guard` reads it via `git log`. A PR template pre-seeds the line. Disable merge/rebase commits; branch protection requires linear history.
- **Alternative rejected:** `COMMIT_MESSAGES` (trailer in a real commit) — concatenates every WIP commit into the body, noisier for the release-time parser. (`PR_BODY` chosen in discussion.)

### D4 — Enforce at PR-time (primary) and release-time (backstop)
Reliability comes from PR-time: a required check on every PR, where the author has context and history is still mutable. The release-time check is the safety net for bypasses (admin override, check misconfig). Relying on release-time alone hits the immutability wall — a trailer-less breaking commit already on `main` can't be auto-corrected.

### D5 — Use a real Conventional-Commits parser
Not a naive `!:` regex. Must read multiline `BREAKING CHANGE:` / `BREAKING-CHANGE:` footers and `!` after an optional scope. Reuse the library release-please uses (`conventional-commits-parser`) or an equivalent, in a small script invoked by the workflow.

### D6 — MIGRATIONS buckets are impact-based, version-agnostic
`## Unreleased — Breaking` / `## Unreleased — Additive` (landed in 53e529a). The guard reads the `Breaking` bucket's `### … — \`<slug>\`` headers for the slug set. Version numbers never appear in `Unreleased` headers — release-please owns the number.

## Risks / Trade-offs

- **[Trailer in PR description is unusual]** → authors forget it. *Mitigation:* PR template scaffolds `Migration:`; the PR-time check fails fast with the exact fix; agents emit it programmatically via `gh pr create --body`.
- **[Squash not actually enforced]** → a merge commit slips through and the trailer/PR-title assumptions break. *Mitigation:* repo settings disable the other merge methods AND branch protection requires linear history (belt + suspenders).
- **[release-please's own commit looks like a change]** → self-blocking releases. *Mitigation:* exclude commits authored by the release-please bot / matching `^chore\(main\): release`.
- **[Multi-breaking-change PR]** → one squash commit, several slugs. *Mitigation:* allow repeated `Migration:` trailers or a comma list; the parser collects all.
- **[Revert of a breaking change]** → `revert:` commit. *Mitigation:* treat as needing `Migration: none — revert of <sha>` (documented in the ADR).
- **[False-positive blocks erode trust]** → people rubber-stamp. *Mitigation:* the `none — <reason>` escape hatch is first-class.

## Migration Plan

1. ADR + finalize the MIGRATIONS maintenance note to match.
2. PR template with the `Migration:` line.
3. `migration-guard` (PR-time) workflow + guard script; make it a required check.
4. Repo settings (squash-only, `PR_BODY`) + branch protection — applied by a maintainer (`gh api` commands in the ADR).
5. `migration-release-guard` (release-time) workflow/job next to release-please.
6. (Deferred) auto-graduation at release.

**Rollback:** the checks are additive CI; disabling the workflows / dropping the required-check setting reverts with no code impact.

## Open Questions

- Guard script language — Go (consistent with the repo, but heavier) vs a small Node script reusing `conventional-commits-parser` (lighter, matches release-please's own stack). Leaning Node.
- Does the release-time check run as a job in the existing release-please workflow, or a separate workflow triggered on the release-please PR? (Prefer a job in the release PR's checks so it blocks the merge that tags.)
